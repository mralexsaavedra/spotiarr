# AGENTS.md

Instructions for AI coding assistants working in `spotiarr`.

> **Single source of truth.** `CLAUDE.md` and `GEMINI.md` are symlinks generated by `skills/setup.sh`.
> Skills live in `skills/` — load the relevant skill BEFORE writing any code.
> Per-workspace guidance lives in `apps/<app>/AGENTS.md` and `packages/<pkg>/AGENTS.md` — workspace docs override root on conflict.

---

## Project Overview

Spotiarr is a self-hosted Spotify companion for tracking and downloading music releases.

| Component | Location          | Tech Stack                                            |
| --------- | ----------------- | ----------------------------------------------------- |
| Backend   | `apps/backend`    | Node 22, Express, Prisma (SQLite), BullMQ, Redis, Zod |
| Frontend  | `apps/frontend`   | React 19, Vite, TanStack Query, Zustand, Tailwind 4   |
| Shared    | `packages/shared` | DTOs, enums, shared utilities                         |

External services: **Redis**, **FFmpeg**, **yt-dlp**, Python 3.11/3.12.

## 2) Agent Checklist

**Before starting:**

- Confirm scope: backend, frontend, shared, or cross-workspace.
- Keep changes minimal and focused on the objective.
- Do not edit unrelated files.
- Never commit secrets (`.env`, tokens, credentials).
- Use `gh` for GitHub operations (PRs, checks, comments).
- Preserve layered architecture and controllers/hooks patterns.
- Load the relevant skill(s) from `skills/` before writing code.

**Before finishing:**

- Run targeted validation for affected workspaces (see §6).
- If the change is broad, run full-repo validation.
- Update docs if flows, commands, or conventions changed.

## 5) Core Conventions

### Formatting (Prettier)

```
semi: true
singleQuote: false
trailingComma: "all"
printWidth: 100
tabWidth: 2
useTabs: false
arrowParens: "always"
```

Import order (both frontend and backend): third-party → `@/` aliases → relative.

Pre-commit hooks run lint + format automatically via `lint-staged` + Husky. Never skip with `--no-verify`.

### TypeScript

- Strict mode enabled (`strict: true`) in all workspaces.
- Prefer `import type` for type-only imports.
- Use explicit domain types over `any`.
- Shared DTOs and enums live in `packages/shared` — never duplicate in apps.

### Architecture

- Backend follows layered architecture: routes → controllers → services → repositories.
- Frontend follows container/presentational split with TanStack Query for server state and Zustand for client state.
- Cross-cutting changes always touch `packages/shared` first, then consumers.

## 6) Validation Commands

Run targeted first, broad only for cross-workspace changes.

| Scope                   | Commands                                                                               |
| ----------------------- | -------------------------------------------------------------------------------------- |
| Frontend only           | `pnpm --filter frontend run lint` → `pnpm --filter frontend run build`                 |
| Backend only            | `pnpm --filter backend run lint` → `pnpm --filter backend run build`                   |
| Shared package          | `pnpm --filter @spotiarr/shared run lint` → `pnpm --filter @spotiarr/shared run build` |
| Broad / cross-workspace | `pnpm lint` → `pnpm build`                                                             |

## 7) Git and PR Policy

- Branch naming: `feature/<slug>`, `fix/<slug>`, `docs/<slug>`, `refactor/<slug>`, `chore/<slug>`.
- Commit messages must follow **Conventional Commits**: `<type>(<scope>): <subject>`.
  - Types: `feat`, `fix`, `docs`, `style`, `refactor`, `chore`, `perf`, `test`.
  - Example: `feat(backend): add download retry with exponential backoff`
- Do not add AI signature footers or `Co-Authored-By` lines.
- Use `gh pr create` for pull requests; include a clear description of _why_, not just _what_.
- Never push directly to `main`.
- Pre-commit hooks (`lint-staged`) must pass before commit. Do not bypass with `--no-verify`.

---

## 8) SDD Delivery Gate

For medium/large changes, use the SDD flow (`/sdd-new`, `/sdd-ff`). Full skill listing: `skills/`.

After `/sdd-verify` completes:

- No `CRITICAL` issues → open PR with `spotiarr-workflow` skill.
- `CRITICAL` issues present → fix before PR. Do not proceed.
- Only `WARNING` / `SUGGESTION` → document in PR description, proceed.

## 9) Multi-Tool Compatibility

Supported: Claude Code, Opencode, Codex, Gemini CLI, GitHub Copilot.

Skills are committed at `skills/`. Per-tool symlinks (`CLAUDE.md`, `GEMINI.md`, `.claude/skills`, `.agents/skills`, etc.) are **local-only** and gitignored. Generate them after cloning:

```bash
bash skills/setup.sh --all
```

See `skills/setup.sh --help` for per-tool options.
